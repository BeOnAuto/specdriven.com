# Community

Spec-Driven Development is bigger than any one tool, team, or dialect. It's a growing movement of practitioners who believe that specifications deserve to be first-class citizens in software development.

## Contribute to This Site

This site is open source. Every page has an "Edit this page on GitHub" link at the bottom. Here are some ways you can help:

- **Fix something:** Typo, broken link, unclear section? Submit a pull request.
- **Add a tool profile:** Building a spec-driven tool? Add it to the [landscape](/landscape/).
- **Add a person:** Know someone who should be on the [notable people](/people) page?
- **Add a timeline entry:** Something missing from the [history](/timeline)?
- **Add a resource:** Books, talks, articles for the [resources](/resources) page.
- **Discuss the evaluation framework:** Think a [dimension](/landscape/#evaluation-framework) is missing or mislabeled? Open an issue.

[View the repo on GitHub →](https://github.com/BeOnAuto/specdriven.com)

## Submit a Spec Dialect

Are you building a structured specification language? We want to feature it.

A spec dialect is more than a workshop technique or a meeting format. Our working definition is that it should have:

- **A structured format** with a defined schema or grammar
- **Machine-parseable syntax** that tooling can process
- **A path to execution**: a relationship to tests, code generation, or validation

This definition is evolving. If you think it should be refined, that's a conversation worth having. Open an issue on the [GitHub repo](https://github.com/BeOnAuto/specdriven.com), submit a pull request, or come discuss it on Discord. Spec-Driven Development is a community effort, and so is defining what belongs in it.
